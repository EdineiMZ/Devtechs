A auditoria abrangeu análise estática de código, revisão de configurações de infraestrutura, teste funcional de todos os módulos e avaliação de mecanismos de autenticação e autorização.

A aplicação possui uma base sólida de segurança — JWT com rotação, rate limiting por IP, auditoria de eventos, guards de permissão em todos os endpoints sensíveis, e proteção contra open redirect. Os principais riscos identificados concentram-se em gestão de segredos e hardening de tokens de sessão.

Achados Críticos

CRÍTICO SEC-001 — Credenciais reais expostas no arquivo .env
.env (linhas 87–88, 94, 105–106)
O arquivo .env contém tokens e senhas de produção que, embora estejam no .gitignore, ficam expostos a qualquer pessoa com acesso à máquina de desenvolvimento ou que obtenha acesso ao ambiente:
  • Mercado Pago: MP_ACCESS_TOKEN=TEST-6479766634240109-…
  • Resend API: RESEND_API_KEY=re_iAJSLi4A_…
  • Hostinger API: HOSTINGER_API_TOKEN=y3PD5OcOEUOAZ34U…
  • PostgreSQL: senha com caracteres especiais em texto claro
Remediação imediata

1. Revogar e regenerar todos os tokens listados acima.
2. Migrar para gerenciador de segredos (AWS Secrets Manager, HashiCorp Vault, ou Doppler).
3. Para desenvolvimento: usar .env.local (gitignored separadamente) com variáveis injetadas por CI/CD.
4. Adicionar git-secrets ou gitleaks como hook de pre-commit.

Achados de Alta Severidade

ALTO SEC-002 — Refresh Token exposto ao cliente JavaScript
apps/web/src/auth.ts (linhas 263, 319)
O refresh token é armazenado no objeto de sessão e retornado via useSession(), tornando-o acessível ao JavaScript do lado do cliente. Um XSS bem-sucedido pode roubar o refresh token e manter acesso persistente mesmo após expiração do access token.
Remediação

Remover session.refreshToken do retorno da callback session(). Armazenar o refresh token exclusivamente no JWT HttpOnly do servidor. Criar um endpoint POST /api/auth/refresh que leia o token via cookie e execute o refresh transparentemente.

ALTO SEC-003 — Segredos JWT fracos / previsíveis em desenvolvimento
.env (linhas 32–38)
Os segredos JWT usam strings legíveis como dev-jwt-access-secret-please-change-in-prod-2026-04. Mesmo em desenvolvimento, segredos previsíveis reduzem a eficácia de testes de segurança locais e aumentam o risco de "copy-paste" para produção.
Remediação

Gerar segredos com: openssl rand -base64 64 para cada variável JWT.
Adicionar validação de startup que rejeite segredos menores que 32 caracteres em produção.

ALTO SEC-004 — Swagger/OpenAPI exposto sem autenticação de camada de rede
infra/nginx.conf (linhas 126–140)
A documentação Swagger dos serviços (/auth/docs, etc.) é servida pelo nginx sem autenticação adicional. Mesmo que a aplicação desabilite Swagger em produção via variável de ambiente, a configuração nginx não tem fallback.
Remediação

Adicionar satisfy any; allow 127.0.0.1; deny all; no bloco nginx para rotas /docs em produção. Alternativamente, usar auth_request para verificar sessão válida antes de servir a documentação.

ALTO SEC-005 — Valores placeholder no .env.example muito próximos de credenciais reais
.env.example
O arquivo de exemplo usa valores como POSTGRES_PASSWORD=devtechs e minioadmin para MinIO — credenciais padrão amplamente conhecidas. Desenvolvedores novos podem copiá-las sem alterar.
Remediação

Substituir por POSTGRES_PASSWORD=SUBSTITUA_POR_SENHA_FORTE_AQUI e adicionar comentários explicando requisitos mínimos (ex.: mínimo 24 caracteres, gerado por openssl rand).

ALTO SEC-006 — Rate limiting usando memória local (não Redis)
services/auth-service/src/common/rate-limit/rate-limit.module.ts
O módulo de throttling global cai para armazenamento em memória por instância. Em deployments multi-instância (horizontal scaling), cada instância mantém contadores independentes, permitindo que atacantes ultrapassem limites distribuindo requisições entre instâncias. O LoginRateLimitGuard (Redis-backed) é a exceção correta.
Remediação

Implementar ThrottlerStorageRedisService via @nest-lab/throttler-storage-redis para todos os limitadores de taxa. O pattern já existe no LoginRateLimitGuard e pode ser replicado.

Achados de Média Severidade

MÉDIO SEC-007 — Content Security Policy (CSP) ausente
apps/web/next.config.ts / infra/nginx.conf
A aplicação define headers como X-Frame-Options e X-Content-Type-Options, mas não possui Content-Security-Policy. Scripts inline e eval() ficam não-restritos, ampliando superfície de ataque XSS.
Remediação

Adicionar em next.config.ts: 

{ key: 'Content-Security-Policy',
  value: "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' ws: wss:;" }
Iniciar com política permissiva e endurecer progressivamente usando relatórios (report-uri).

MÉDIO SEC-008 — CORS com exposedHeaders: Authorization
services/*/src/main.ts
A configuração CORS expõe o header Authorization para todas as origens permitidas. Isso permite que páginas em origens listadas leiam tokens Bearer de respostas cross-origin.
Remediação

Remover Authorization de exposedHeaders. O cliente já possui o token no momento da requisição; não é necessário lê-lo da resposta.

MÉDIO SEC-009 — Ausência de @UseGuards explícito em alguns controllers
services/notification-service/src/ · services/payments-service/src/
Alguns controllers dependem implicitamente do guard global registrado no AppModule. Refatorações futuras que removam o guard global podem expor endpoints involuntariamente.
Remediação

Adicionar @UseGuards(JwtAuthGuard) explicitamente em cada controller que requer autenticação. Defesa em profundidade.

MÉDIO SEC-010 — Porta 5432 do PostgreSQL exposta no host
infra/docker-compose.yml (linha 55)
O PostgreSQL expõe a porta 5432 em todas as interfaces do host. Em ambiente de desenvolvimento com redes compartilhadas ou VMs, isso aumenta a superfície de ataque.
Remediação

Alterar para 127.0.0.1:5432:5432 para restringir ao localhost, ou remover o mapeamento de porta e acessar exclusivamente via rede Docker interna.

MÉDIO SEC-011 — Ausência de rotação de refresh token (token reuse detection)
apps/web/src/auth.ts (linhas 268–288)
O fluxo de refresh emite novos tokens mas não invalida o refresh token anterior, permitindo replay de refresh tokens roubados.
Remediação

Implementar "refresh token family" — ao usar um refresh token, marcá-lo como usado e detectar reuso (indicador de roubo). Invalidar toda a família de tokens em caso de reuso.

MÉDIO SEC-012 — Credenciais padrão em docker-compose
infra/docker-compose.yml (linhas 50–53)
Valores default como POSTGRES_USER:-devtechs e POSTGRES_PASSWORD:-devtechs são usados quando variáveis não estão definidas.
Remediação

Remover os fallbacks e exigir variáveis explicitamente. Adicionar validação no startup que falhe se variáveis obrigatórias estiverem ausentes.

MÉDIO SEC-013 — Autenticação entre serviços (service-to-service) incompleta
services/* (múltiplos)
Existe AUTH_INTERNAL_SECRET mas não é utilizado de forma consistente em todas as chamadas inter-serviços. Serviços comprometidos poderiam fazer chamadas não-autenticadas.
Remediação

Implementar um InternalAuthGuard que verifique o header X-Internal-Secret em todas as rotas inter-serviços. Usar o AUTH_INTERNAL_SECRET já definido no .env.

MÉDIO SEC-014 — Limite de body size inconsistente (nginx vs. serviços)
infra/nginx.conf (linha 30) · services/payments-service/src/main.ts
Nginx permite client_max_body_size 25m enquanto serviços limitam a 1MB. A discrepância pode mascarar vulnerabilidades de upload em ambientes sem nginx.
Remediação

Alinhar limites por rota: 10mb para uploads de documentos, 100kb para endpoints de API padrão. Definir no nginx por location block.

MÉDIO SEC-015 — Eventos de segurança não auditados completamente
services/auth-service/src/modules/audit/
Tentativas de refresh de token, bypasses de rate limit, e tentativas de escalação de permissão não são registradas no log de auditoria.
Remediação

Expandir o AuditDecorator para cobrir: refresh token usage, rate limit hits, permission denied, e OAuth account linking.

MÉDIO SEC-016 — HTTP sem redirecionamento para HTTPS em nginx
infra/nginx.conf
O servidor HTTP (porta 80) não redireciona para HTTPS, tornando cookies de sessão vulneráveis a interceptação em produção.
Remediação

Adicionar no bloco server porta 80: return 301 https://$host$request_uri; e garantir que cookies de sessão tenham atributo Secure.

MÉDIO SEC-017 — CSRF não documentado para rotas de API customizadas
apps/web/src/app/api/
NextAuth v5 protege suas rotas automaticamente, mas rotas de API customizadas não possuem proteção CSRF explícita documentada.
Remediação

Verificar que todas as rotas POST/PUT/DELETE customizadas exigem sessão válida via auth(). NextAuth's session cookie é SameSite=Lax por padrão — suficiente para a maioria dos casos. Documentar a estratégia.

Achados de Baixa Severidade

BAIXO SEC-018 — Header X-Permitted-Cross-Domain-Policies ausente
apps/web/next.config.ts
Proteção contra políticas cross-domain (Flash/PDF) ausente.
Remediação

Adicionar: { key: 'X-Permitted-Cross-Domain-Policies', value: 'none' }

BAIXO SEC-019 — Endpoint /health sem rate limiting
services/*/src/health.controller.ts
Endpoints de health check estão sem rate limiting. Embora de baixo risco, podem ser alvos de fingerprinting de serviços.
Remediação

Limitar /health a 60 requisições/minuto por IP. Adequado para health checks de load balancers.

BAIXO SEC-020 — Logs de desenvolvimento podem vazar informações de stack traces
services/*/src/main.ts (LOG_LEVEL=info)
Com LOG_LEVEL=info, stack traces completos de erros podem ser retornados em respostas de erro em ambiente de desenvolvimento.
Remediação

Em produção, usar LOG_LEVEL=warn e garantir que respostas de erro retornem apenas mensagens genéricas para o cliente (o NestJS HttpExceptionFilter já faz isso — verificar configuração).
