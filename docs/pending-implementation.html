1. Módulo de Chamados (chamados + chat real-time)

P0 — Crítico Frontend Next.js (apps/web) Backend já existe

Contexto

O support-service em :4008 já expõe REST (CRUD de tickets, mensagens, atribuição, SLA) e um gateway Socket.IO no namespace /support com eventos ticket:join, message:send, message:new, typing:start/stop, user:typing, user:joined, ticket:status. Isso foi verificado em sessão E2E — o cliente conecta com JWT, recebe connected com isAgent resolvido a partir de support:tickets:close.

Já existem no front: o cliente Socket.IO tipado em apps/web/src/lib/support-socket.ts e o hook useTicketChat em apps/web/src/hooks/use-ticket-chat.ts. O que falta é a UI que consome tudo.

Escopo

  • Lista de chamados em /perfil/tickets: filtros por status, busca, paginação, badge de SLA.
  • Criação em /perfil/tickets/novo: formulário com título, categoria, prioridade, descrição, anexos.
  • Detalhe + chat em /perfil/tickets/[id]: timeline de mensagens, indicador de digitando, envio em tempo real, botão "marcar como resolvido", histórico de status, atribuição (admin), notas internas (somente agentes).
  • Visão admin/agente em /admin/suporte: queue de tickets atribuídos/não atribuídos, ações em massa, base de conhecimento (KB), métricas de SLA.

Arquivos a criar

  • apps/web/src/app/perfil/tickets/page.tsx — server component, lista
  • apps/web/src/app/perfil/tickets/novo/page.tsx — formulário criação
  • apps/web/src/app/perfil/tickets/[id]/page.tsx — detalhe (server) + chat (client)
  • apps/web/src/app/perfil/tickets/[id]/ticket-chat.tsx — client component com useTicketChat
  • apps/web/src/app/admin/suporte/page.tsx — queue de agente
  • apps/web/src/app/admin/suporte/[id]/page.tsx — detalhe agente (com notas internas)
  • apps/web/src/lib/support-api.ts — wrapper REST tipado para o support-service
  • apps/web/src/components/tickets/* — TicketCard, TicketStatusBadge, MessageBubble, TypingIndicator, NewMessageComposer, TicketFilters

Prompt

Você está no monorepo DevTechs (Turborepo + pnpm) na branch atual. Implemente o módulo de Chamados (tickets) na app `apps/web` consumindo o `services/support-service` que já existe e está rodando em `http://127.0.0.1:4008`.

CONTEXTO TÉCNICO QUE JÁ EXISTE (NÃO REIMPLEMENTAR):
- `apps/web/src/lib/support-socket.ts` — fábrica `createSupportSocket(accessToken)` tipada com eventos cliente/servidor.
- `apps/web/src/hooks/use-ticket-chat.ts` — hook que monta o socket, faz `ticket:join`, expõe `messages`, `typers`, `sendMessage(body, isInternal?)`, `notifyTyping()`, status de conexão.
- `services/support-service/src/modules/chat/support.gateway.ts` — gateway Socket.IO no namespace `/support` autenticado por JWT no handshake (`auth.token`).
- `services/support-service/src/modules/tickets/*` — controller REST com CRUD de tickets e mensagens. Inspecione os DTOs e endpoints reais antes de codar a camada de fetch.
- Sessão NextAuth em `apps/web/src/auth.ts` — `session.accessToken` é o JWT que deve ir para o socket e para o header `Authorization: Bearer ...` nos fetches REST.
- Permissões: `support:tickets:view` (leitura), `support:tickets:close` (resolver/agente), `support:tickets:manage`, `support:tickets:assign`, `support:kb:edit`. A presença de `support:tickets:close` no `session.user.permissions` define se a UI deve mostrar recursos de agente.

On submit: `createTicket` → redirect `/perfil/tickets/`. Erros de campo inline.
   - `[id]/page.tsx` (server component): valida sessão + propriedade do ticket (cliente só vê seus próprios; agente vê todos). Carrega ticket + mensagens iniciais via REST e passa para ``.

5. Atualizar `apps/web/src/components/app/nav-config.tsx` — confirmar que os hrefs `/perfil/tickets` e `/admin/suporte` já apontam para os novos arquivos (eles já estão no nav, basta deixar os arquivos no lugar certo).

REGRAS GERAIS:
- Use Tailwind + componentes de `@devtechs/ui` (Button, Input, Badge, Card etc.) — siga o padrão das páginas existentes `/perfil/page.tsx` e `/admin/page.tsx`.

- Toda chamada server-side ao `support-service` deve usar `cache: 'no-store'` para não cachear dados sensíveis em build.
- Trate fail-soft: se o socket cair (Redis offline, support-service reiniciando), a UI mostra um banner discreto "Conexão perdida — tentando reconectar…" sem quebrar a página.

- Antes de finalizar, suba a app (`pnpm dev:web`) e valide manualmente: criar ticket → enviar mensagem → ver chegando em tempo real numa segunda aba do mesmo browser.

2. Auth gate nos apps de módulo (rh, financeiro, projetos, devops, suporte, store, developer)

P0 — Crítico (segurança) 7 apps Next.js Decisão arquitetural

Contexto

O requisito é "somente a página inicial pública; o restante exige login". Hoje os apps :3001 a :3007 servem suas raízes anonimamente (são stubs <main>DevTechs - X app</main>). Eles não compartilham o cookie de sessão do app web (origens diferentes), então cada um precisa validar a sessão via auth-service ou ser substituído por sub-rotas dentro do web.

Decisão arquitetural recomendada: consolide tudo em apps/web com sub-rotas (/admin/<modulo>) e descarte os apps módulo separados — a ergonomia de uma única origem é muito melhor para autenticação, navegação e SEO. Os apps módulo só fazem sentido se forem deploys independentes em domínios distintos (ex.: rh.devtechs.com), o que não é o caso hoje.

Prompt

No monorepo DevTechs, hoje os apps Next.js em `apps/{rh,financeiro,projetos,devops,suporte,store,developer}` rodam em portas separadas (3001-3007) e estão acessíveis sem nenhum gate de autenticação. Eles também são stubs (`
DevTechs - X app
`).

OBJETIVO: garantir que apenas a página inicial pública do app `web` (`:3000` rota `/`) seja anônima. Toda outra rota exige sessão NextAuth válida com email verificado.

Há duas abordagens. Avalie e, ao final, IMPLEMENTE A ABORDAGEM A (consolidação) — é a mais aderente ao restante do código:

ABORDAGEM A (recomendada — consolidar em apps/web):
1. Mover todas as funcionalidades dos apps módulo para sub-rotas dentro de `apps/web/src/app/admin//`. As implementações concretas estão em outros prompts deste documento.
2. Atualizar `apps/web/src/middleware.ts` matcher para incluir os novos prefixos: `/admin/:path*` já está no matcher; basta confirmar que `/perfil/:path*` também cobre todas as sub-rotas do cliente.
3. Remover do `pnpm-workspace.yaml` os pacotes `@devtechs/{rh,financeiro,projetos,devops,suporte,developer}` (manter `store` se ele for um SaaS público de planos — confirmar com o time).
4. Remover os diretórios `apps/{rh,financeiro,...}` e os scripts `dev:*` do `package.json` raiz que apontavam para eles.
5. Atualizar `infra/nginx.conf` removendo os `location /rh/`, `/financeiro/` etc.; tudo passa a vir de `apps/web` que serve `/admin/` e `/perfil/`.
6. Atualizar README.md descrevendo a nova estrutura.
7. Validar com `pnpm --filter @devtechs/web build` e rodar Playwright E2E confirmando que `http://localhost:3001/` etc. param de responder.

ABORDAGEM B (manter apps separados — só faça se houver requisito explícito de deploy independente):
1. Em CADA app módulo, criar `src/middleware.ts` que valida a sessão chamando `GET /auth/me` no `auth-service` (lendo o cookie `authjs.session-token` ou um header `Authorization` repassado pelo nginx). Como o cookie é por-origem, a app principal precisa setar um cookie de domínio mais amplo (`.devtechs.local`) para compartilhar — isso muda a configuração do NextAuth (`cookies.sessionToken.options.domain`).
2. Cada app passa a fazer redirect para `http://localhost:3000/login?callbackUrl=` quando não autenticado.
3. Adicionar testes que confirmam o redirect em cada app.

ENTREGÁVEIS COMUNS:
- `scripts/e2e-tests/04-modules.mjs` atualizado para refletir a nova estrutura (rotas dentro de `:3000` em vez de portas separadas).
- Documentação no README explicando a decisão.
- Auditoria de segurança: rodar `scripts/e2e-tests/07-security-audit.mjs` e confirmar que nenhuma rota antes acessível anonimamente continua acessível.

Cada sub-rota tem uma permissão obrigatória. No topo de cada `page.tsx`:
```ts
const session = await auth();

const has = (perm: string) => session.user.permissions.includes(perm);
if (!has(REQUIRED_PERM)) redirect('/perfil');
```

Prompt

CONDICIONAL — só execute este prompt se a decisão for manter `apps/{rh,financeiro,projetos,devops,suporte,developer}` como apps Next.js separadas. Caso contrário, delete os diretórios conforme prompt 2.

PRÉ-REQUISITO ARQUITETURAL: configurar cookie de sessão NextAuth com domínio `.devtechs.local` (ou um domínio compartilhado) em `apps/web/src/auth.ts`:
```ts
cookies: {
  sessionToken: {
    name: 'authjs.session-token',
    options: {
      httpOnly: true,
      sameSite: 'lax',
      secure: process.env.NODE_ENV === 'production',
      path: '/',
      domain: process.env.AUTH_COOKIE_DOMAIN, // .devtechs.local

  },
},
```
