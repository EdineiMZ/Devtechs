3. Correções Realizadas

Fix #1 — Imports ausentes em 4 serviços

BUG

Os serviços finance-service, notification-service, payments-service e license-service tinham main.ts sem os imports necessários para exportação OpenAPI, causando falha de compilação TypeScript.

services/finance-service/src/main.ts

services/notification-service/src/main.ts

services/payments-service/src/main.ts

services/license-service/src/main.ts

// Imports adicionados em cada main.ts:
import { writeFileSync, mkdirSync } from 'node:fs';
import { dirname } from 'node:path';
import { setupSwagger } from './swagger';

Fix #2 — license-service deleteOutDir + stale tsbuildinfo

BUG

nest-cli.json tinha deleteOutDir: true causando deleção do dist/ seguido de falha silenciosa de recompilação por tsconfig.tsbuildinfo desatualizado.

services/license-service/nest-cli.json

// Antes (com problema):
{ "deleteOutDir": true, "sourceRoot": "src" }

// Depois (corrigido):
{ "$schema": "...", "collection": "@nestjs/schematics", "sourceRoot": "src" }

Fix #3 — Token de verificação de email (formato inválido)

BUG CRÍTICO

auth.service.ts usava randomUUID() para gerar o token de verificação, mas o DTO VerifyEmailQueryDto valida @IsHexadecimal() + @Length(64,64). UUID não é hexadecimal de 64 chars → verificação impossível.

services/auth-service/src/modules/auth/auth.service.ts:1,90

// Antes (com problema):
import { randomUUID } from 'node:crypto';
token: randomUUID(),

// Depois (corrigido):
import { randomBytes, randomUUID } from 'node:crypto';
token: randomBytes(32).toString('hex'), // 64 chars hex = correto

Fix #4 — Permissões ausentes no banco de dados

CONFIG

Os serviços usavam permission keys nos guards que não existiam no banco de dados. Os endpoints retornavam 403 mesmo para o admin. 15 novas permissões foram criadas e atribuídas ao role admin.

Achados Críticos

SEC-001 — Token de API Hostinger exposto no .env

CRÍTICO

.env:linha ~98

Token HOSTINGER_API_TOKEN ativo exposto em arquivo de texto plano no repositório. Permite acesso não autorizado à conta de hospedagem, modificação de DNS, acesso a VPS.

Ação imediata: Revogar token no painel Hostinger → Gerar novo token → Usar secrets manager (AWS Secrets Manager / Vault) → Purgar do histórico git com BFG Repo-Cleaner.

SEC-002 — Chaves Mercado Pago no .env

CRÍTICO

.env:linhas 79-80

Chaves de teste MP (MP_ACCESS_TOKEN) expostas. Embora sejam de teste, o padrão e estrutura das chaves ficam visíveis, facilitando vazamento das chaves de produção se o mesmo arquivo for reutilizado.

Rotacionar chaves MP → Usar variáveis de ambiente injetadas em CI/CD → Nunca commitar .env real.

SEC-003 — Segredo TOTP em texto plano no Redis durante setup

CRÍTICO

services/auth-service/src/modules/auth/two-factor.service.ts:95-107

Durante o fluxo de configuração 2FA, o segredo TOTP fica armazenado em texto plano no Redis por 10 minutos antes da confirmação do usuário. Acesso ao Redis comprometido → todos os segredos TOTP em setup comprometidos.

Criptografar segredo com AES-256-GCM antes de armazenar no Redis → Usar Redis ACL para restringir acesso por serviço.

Achados de Alta Severidade

SEC-004 — Função timing-safe customizada desnecessária

ALTO

services/auth-service/src/common/guards/internal-secret.guard.ts:62-69

Implementação manual de comparação em tempo constante quando crypto.timingSafeEqual do Node.js existe nativamente. Código customizado aumenta risco de bug sutil.

Substituir por import { timingSafeEqual } from 'node:crypto'.

SEC-005 — Entropia fraca nos códigos de recuperação 2FA

ALTO

services/auth-service/src/modules/auth/two-factor.service.ts:319-321

Códigos de recuperação usam apenas 5 bytes (40 bits) por código. NIST SP 800-63B recomenda mínimo 128 bits de entropia.

// Atual: randomBytes(5) → 40 bits
// Recomendado: randomBytes(16) → 128 bits

SEC-006 — Cache de permissões não invalidado em mudanças de role

ALTO

services/auth-service/src/common/guards/permission.guard.ts

Invalidação do cache (TTL 5 min) é responsabilidade do chamador, mas não há enforcement. Remoção de permissão de usuário permanece ativa por até 5 minutos.

Implementar invalidação automática em RolesService via Redis pub/sub ao modificar roles/permissões.

SEC-007 — Race condition no refresh token rotation

ALTO

services/auth-service/src/modules/auth/auth.service.ts:325

Duas requisições concorrentes podem ambas verificar o token como válido antes de qualquer uma persistir a revogação, permitindo uso duplicado do refresh token.

Usar UPDATE ... WHERE refreshToken = $token RETURNING id com SELECT FOR UPDATE, ou Redis distributed lock (SET NX).

SEC-008 — Logout não invalida outras sessões do mesmo usuário

ALTO

services/auth-service/src/modules/auth/auth.service.ts:365

Logout revoga apenas a sessão atual. Em cenário de dispositivo roubado, outras sessões ativas permanecem válidas.

Adicionar endpoint POST /auth/logout-all que revoga todas as sessões do usuário autenticado.

Achados de Média Severidade

SEC-009 — SQL raw no AuditService

MÉDIO

services/auth-service/src/modules/audit/audit.service.ts:253-315

Uso de $queryRaw com template literals do Prisma (parametrizado e seguro), mas aumenta superfície de ataque e dificulta auditoria.

Preferir Prisma Query API; documentar explicitamente por que raw SQL é necessário neste caso.

SEC-010 — CORS sem maxAge no preflight

MÉDIO

Todos os serviços configuram CORS sem maxAge, forçando preflight a cada requisição. Sem cache de preflight, ataques de flooding OPTIONS são facilitados.

Adicionar maxAge: 3600 ao app.enableCors() em todos os serviços.

SEC-011 — Swallow de erros Redis em modo desenvolvimento

MÉDIO

services/notification-service/src/main.ts:12-39

Handler uncaughtException/unhandledRejection em modo dev suprime erros Redis, podendo mascarar exaustão de pool ou ataques de DoS à conexão Redis.

Remover supressão de erros. Usar circuit breaker (e.g., opossum) para Redis em vez de silenciar.

SEC-012 — Ausência de timeout em processamento de webhooks

MÉDIO

services/payments-service/src/modules/webhooks/

Handlers de webhook sem timeout explícito. Payload malicioso ou API externa lenta pode bloquear o event loop por tempo indeterminado.

Adicionar timeout de 5s ao processamento; publicar evento assíncrono e retornar 200 imediatamente ao Mercado Pago.

Achados de Baixa Severidade

SEC-013 até SEC-020 — Headers de segurança ausentes

BAIXO

infra/nginx.conf | apps/web/src/middleware.ts

  • Ausência de Strict-Transport-Security (HSTS)
  • Ausência de X-Content-Type-Options: nosniff
  • Ausência de X-Frame-Options: DENY
  • Ausência de Content-Security-Policy
  • Ausência de Referrer-Policy
  • client_max_body_size 25m muito permissivo
  • Access token completo armazenado na tabela de sessões
  • Swagger acessível em produção se EXPOSE_SWAGGER_IN_PROD=true
Adicionar headers de segurança ao nginx.conf e ao middleware Next.js. Armazenar apenas hash do access token na sessão.

Pontos Positivos (Informativo)

✅ Implementações Exemplares de Segurança

  • Rate limiting sofisticado no login: 5 tentativas/15min → bloqueio de 1h (LoginRateLimitGuard)
  • 2FA com token temporário: 5 minutos de validade + campo typ: 'awaiting2FA' — previne cross-type reuse
  • Criptografia de segredos TOTP: AES-256-GCM com IV aleatório
  • Bcrypt rounds = 12: Configuração forte para hashing de senhas
  • Audit logging abrangente: Todas as ações sensíveis logadas com IP e user-agent
  • Refresh token rotation: Detecção de reutilização com revogação defensiva
  • Email verification token: 64 chars hexadecimais (256 bits de entropia)
  • Identical error messages: "Invalid credentials" para usuário não encontrado e senha errada (anti-enumeração)

9. Roadmap de Mitigação

Fase 1 — Imediato (24-48h)

  • 🔴 Revogar token Hostinger e gerar novo
  • 🔴 Rotacionar chaves Mercado Pago
  • 🔴 Remover .env do histórico git (BFG Repo-Cleaner)
  • 🔴 Adicionar .env ao .gitignore definitivamente
  • 🔴 Migrar segredos para AWS Secrets Manager ou HashiCorp Vault

Fase 2 — Curto prazo (1-2 semanas)

  • 🟠 Criptografar segredo TOTP antes do Redis
  • 🟠 Aumentar entropia de recovery codes para 128 bits
  • 🟠 Implementar invalidação automática do cache de permissões
  • 🟠 Adicionar maxAge ao CORS preflight
  • 🟠 Substituir timingSafeCompare por crypto.timingSafeEqual

Fase 3 — Médio prazo (1 mês)

  • 🟡 Distributed lock para refresh token rotation
  • 🟡 Endpoint POST /auth/logout-all
  • 🟡 Adicionar headers de segurança ao nginx (HSTS, CSP, X-Content-Type-Options)
  • 🟡 Timeout em webhooks do Mercado Pago
  • 🟡 Remover supressão de erros Redis em desenvolvimento

Fase 4 — Contínuo

  • 🟢 Auditoria trimestral de dependências (pnpm audit)
  • 🟢 Pentest mensal nos endpoints de autenticação
  • 🟢 Alertas SIEM para tentativas de 2FA falhas > 3/hora
  • 🟢 Revisão do acesso Redis ACL por serviço
  • 🟢 Scan de segredos no CI/CD (detect-secrets ou gitleaks)
